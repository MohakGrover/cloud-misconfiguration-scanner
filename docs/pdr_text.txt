Product Design & Requirements Document (PDR)
Multi-Cloud Misconfiguration Security Scanner
1. Product Vision & Problem Statement
Vision
Build a lightweight, education-focused Cloud Security Posture Management (CSPM) tool that empowers
security learners and small teams to detect and understand cloud misconfigurations across AWS, Azure, and
GCP through automated scanning, clear reporting, and hands-on validation environments.
Problem Statement
Cloud environments are increasingly complex, and misconfigurations remain the leading cause of cloud security
breaches. Existing CSPM solutions are either:
Enterprise-focused with steep learning curves and high costs
Lacking multi-cloud support
Opaque in their detection logic, making learning difficult
Target Problem: Security engineers, cloud practitioners, and learners need an accessible, transparent tool to:
Identify common cloud misconfigurations across multiple providers
Understand why configurations are risky
Practice detection in safe lab environments
Learn CSPM architecture patterns

3. Core User Journeys
Journey 1: First-Time Setup & Scan
1. User clones repository and installs dependencies
2. User configures cloud provider credentials (AWS/Azure/GCP)
3. User runs first scan via CLI: python scan.py --provider aws
4. System collects cloud configurations via APIs
5. System runs CIS benchmark rules against collected data
6. User views findings in terminal output (summary + severity)
7. User opens web dashboard to see detailed findings
8. User filters by severity, service, or compliance framework

--- Page 3 ---
Success Criteria: User can run their first scan and view results within 10 minutes
Journey 2: Scanning Infrastructure-as-Code (IaC)
1. User points scanner at Terraform files directory
2. User runs: python scan.py --iac --path ./terraform
3. System parses .tf files and extracts resource configurations
4. System applies security rules to IaC definitions
5. User sees "shift-left" findings before deployment
6. User reviews specific file/line recommendations
7. User fixes issues in Terraform code
Success Criteria: User identifies IaC misconfigurations before cloud deployment
Journey 3: Learning via Validation Lab
1. User enters validation lab environment
2. User deploys intentionally misconfigured resources (via Terraform modules)
3. User runs scanner against lab environment
4. User compares findings with known seeded misconfigurations
5. User explores detection rule logic in codebase
6. User modifies rules to experiment with detection
Success Criteria: User understands how specific misconfigurations are detected
Journey 4: Compliance Reporting
1. User runs scan with compliance flag: python scan.py --compliance cis-aws
2. System maps findings to CIS Benchmark controls
3. User exports compliance report (JSON/CSV/HTML)
4. User shares report with stakeholders
5. User tracks remediation progress over time
Success Criteria: User generates audit-ready compliance report
4. Functional Requirements

FR-1: Cloud Configuration Collection
FR-1.1: Connect to AWS using boto3 with IAM credentials (read-only)
FR-1.2: Connect to Azure using Azure SDK with service principal
FR-1.3: Connect to GCP using Google Cloud Client Libraries
FR-1.4: Collect configurations for supported services:
Storage (S3, Azure Blob, GCS buckets)
Compute (EC2, VMs, GCE instances)
Networking (Security Groups, NSGs, Firewall Rules)
IAM (Policies, Roles, Service Accounts)
Databases (RDS, Azure SQL, Cloud SQL)
FR-1.5: Handle authentication errors gracefully with clear messages
FR-1.6: Support multi-account scanning (multiple credential profiles)
FR-2: Infrastructure-as-Code Scanning
FR-2.1: Parse Terraform (.tf) files recursively from directory
FR-2.2: Extract resource configurations (security groups, IAM, storage, etc.)
FR-2.3: Apply security rules to IaC before deployment
FR-2.4: Report findings with file path and line number
FR-2.5: Support Terraform modules and variables resolution
FR-2.6: AST-Based Parsing (Enhanced):
Move beyond string matching to Abstract Syntax Tree (AST) parsing
Use libraries: python-hcl2 + custom wrapper OR integrate Checkov/KICS engines
Understand Terraform's semantic structure, not just syntax
FR-2.7: Variable Resolution:
Resolve variables from terraform.tfvars , *.auto.tfvars , and locals.tf
Support environment variable interpolation (var.environment )
Handle conditional expressions (count , for_each)
Track variable precedence (CLI > tfvars > defaults)
FR-2.8: Provider Context Mocking:
Mock cloud environment state during IaC evaluation
Example: Evaluate if security group attached to subnet has correct routing

--- Page 5 ---
Simulate resource relationships (VPC → Subnet → SG → Instance)
Detect cross-resource misconfigurations before deployment
FR-2.9: Module Traversal:
Recursively parse Terraform module calls
Resolve module sources (local paths, Terraform Registry, Git)
Track variable flow from root to nested modules
FR-3: Modular Rules Engine
FR-3.1: Define rules as modular, human-readable YAML/JSON files
FR-3.2: Each rule includes:
Rule ID and name
Description and remediation guidance
Severity level (Critical, High, Medium, Low)
CIS Benchmark mapping (e.g., CIS AWS 1.2.0 - Control 4.1)
Detection logic (query/condition)
Parent condition field (optional: skip rule if parent control is enabled)
FR-3.3: Support rule customization (enable/disable, modify thresholds)
FR-3.4: Allow users to add custom rules
FR-3.5: Logic Hierarchy (Parent-Child Rules):
Rules support parent-child dependencies to reduce false positives
Example: If Account_Level_S3_Block_Public_Access  is True, skip individual bucket public access
checks
Parent conditions evaluated before child rules to optimize scan performance
Hierarchical logic documented in rule metadata
FR-3.6: Global Service Registry:
Maintain a registry of global AWS services (IAM, Route53, CloudFront, Organizations)
Flag global services to prevent redundant multi-region API calls
Query global services once per account, not per region
Prevents API throttling and improves scan speed by 40-60%
FR-3.7: Rego Integration (Post-MVP):
Transition from custom YAML to Open Policy Agent (OPA) Rego
Rego as industry-standard policy language (learning objective for Sam)

--- Page 6 ---
Phase 3 feature: Gradual migration path from YAML to Rego
Maintain backward compatibility during transition
FR-4: Misconfiguration Detection
Detect the following misconfiguration categories:
Storage Security:
Public S3/Blob/GCS buckets without encryption
Buckets with public ACLs or policies
Unencrypted storage at rest
Network Security:
Security groups with 0.0.0.0/0 access on sensitive ports (22, 3389, 3306)
Network Security Groups allowing unrestricted inbound traffic
Firewall rules exposing management ports
IAM & Access Control:
Overly permissive IAM roles (e.g., *:* permissions)
Service accounts with excessive privileges
Root account usage without MFA
Access keys older than 90 days
Database Security:
RDS/SQL databases without encryption
Databases publicly accessible
Missing backup configurations
Compute Security:
Instances with public IPs without security groups
Missing security patches/vulnerable AMIs
Unencrypted EBS volumes

FR-5: Validation Lab
FR-5.1: Provide Terraform modules to deploy intentionally misconfigured resources
FR-5.2: Include at least 15 seeded misconfigurations across AWS/Azure/GCP
FR-5.3: Document each seeded issue with expected detection
FR-5.4: Include teardown scripts for easy cleanup
FR-5.5: Tag lab resources to prevent accidental production scanning
FR-5.6: Kill-Switch Tag (Safety Protocol):
MANDATORY tag on every lab resource: ScannerLab: <ISO8601_Timestamp>
Example: ScannerLab: 2026-01-13T14:30:00Z
Prevents lab resources from being mistaken for production
Enables automated identification and cleanup
FR-5.7: Automated Nuke Script:
Standalone Python script: lab_nuke.py
Forcefully deletes ANY resource with ScannerLab tag older than 4 hours
Runs independently of main scanner (can be scheduled via cron)
Multi-cloud support (AWS, Azure, GCP cleanup logic)
Sends summary report: resources deleted, costs prevented
CRITICAL: Build nuke script BEFORE deploy script (safety-first)
FR-5.8: Regional Lockdown:
Lab restricted to single non-production region by default
AWS: us-east-2 (Ohio) - lower cost, non-primary region
Azure: East US 2
GCP: us-central1
Configuration file to override region (with warnings)
FR-5.9: Lab Integrity Verification:
Use terraform plan --detailed-exitcode  to detect drift
Alert if lab resources modified outside Terraform
Prevent "zombie" modifications that break expected findings

5. Non-Functional Requirements
NFR-1: Performance
Scan 100 cloud resources in under 60 seconds
Support scanning up to 1,000 resources without memory issues
Dashboard loads initial view in under 3 seconds
NFR-2: Usability
CLI commands follow intuitive patterns (consistent with industry tools)
Dashboard requires zero configuration to run locally
Error messages are actionable (tell user what to fix)
NFR-3: Security
Store cloud credentials securely (environment variables, no hardcoding)
Use read-only cloud permissions (least privilege)
Sanitize outputs to prevent credential leakage in logs/reports
Validate all inputs (file paths, commands) to prevent injection

--- Page 10 ---
NFR-4: Maintainability
Modular architecture (separation of collectors, rules, UI)
Well-documented code with docstrings
Unit tests for rule logic (80%+ coverage goal)
Easy to add new cloud providers or rules
NFR-5: Portability
Cross-platform (Windows, macOS, Linux)
Containerized deployment option (Docker)
Minimal dependencies (Python 3.9+, standard libraries where possible)
NFR-6: Scalability (Learning Goal)
Architecture supports future multi-threading for parallel scans
Database schema designed for scan history storage (SQLite for MVP)
